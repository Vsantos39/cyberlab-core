# Chapter 01 — Mission

## Mission

The mission of CyberLab 2030 is to transform theoretical knowledge into practical expertise through continuous implementation, documentation, experimentation and automation.

Every concept studied within the project must result in a tangible outcome such as infrastructure, code, documentation, scripts, diagrams or operational procedures.

CyberLab promotes learning by building rather than learning by memorization.

The project values curiosity, discipline, technical quality and continuous improvement above speed.

Its mission is not only to learn technologies, but to understand the principles that make those technologies possible.


# Chapter 02 — The Vision

## Vision

CyberLab 2030 was created with the vision of becoming a permanent learning ecosystem where every new technology can be studied, implemented, documented and improved over time.

Instead of isolated laboratories or temporary projects, CyberLab aims to provide a continuously evolving enterprise environment that reflects real-world infrastructure and cybersecurity scenarios.

The project is designed to grow alongside its creator.

As new technologies emerge, new modules will be incorporated while maintaining the same standards of architecture, documentation, automation and governance.

CyberLab is not expected to have an end.

It is intended to become a lifelong platform for experimentation, professional development and technical excellence.

---

### Long-Term Vision

By 2030, CyberLab should represent a complete enterprise ecosystem capable of simulating modern corporate environments, allowing continuous development in infrastructure, cloud computing, cybersecurity, automation, DevSecOps and governance.

Every technology learned should strengthen the ecosystem rather than exist as an isolated exercise.

# Chapter 04 — Learning Methodology

## Purpose

CyberLab 2030 follows a project-based learning methodology designed to develop practical skills through continuous implementation, documentation, and reflection.

Rather than focusing exclusively on theoretical knowledge, every learning activity is transformed into a tangible project deliverable.

The objective is to develop not only technical competence, but also professional discipline and engineering practices.

---

## The Learning Cycle

Every subject studied within CyberLab follows the same structured cycle.

### 1. Understand

Study the concepts, architecture, and purpose of the technology.

Learning begins by understanding why a technology exists before learning how to use it.

---

### 2. Plan

Design the implementation.

Create diagrams, identify dependencies, define objectives, and document the expected outcome before starting any technical work.

---

### 3. Build

Implement the solution.

Every implementation should follow professional practices, prioritizing organization, security, and reproducibility.

---

### 4. Validate

Test the environment.

Verify that the implementation behaves as expected and identify opportunities for improvement.

Testing is considered part of the learning process.

---

### 5. Document

Record everything that was learned.

Documentation should include architecture decisions, installation procedures, configuration details, troubleshooting steps, lessons learned, and references.

---

### 6. Share

Publish the work through GitHub whenever appropriate.

CyberLab encourages learning in public by maintaining repositories, technical documentation, diagrams, and project history.

---

### 7. Improve

No implementation is considered final.

Every project should evolve through continuous refinement, refactoring, automation, and documentation improvements.

Continuous improvement is part of the methodology.

---

* Practical implementation

Completion is determined by successful delivery rather than course completion.

---

# Learning Philosophy

---

# Knowledge Retention

Knowledge should become part of a permanent technical library.

The CyberLab documentation is intended to become a personal engineering knowledge base that continues growing throughout the professional career.

---

# Continuous Progress

Small, consistent progress is preferred over occasional intensive study sessions.

Every study session, regardless of its duration, should produce a meaningful contribution to the project.

Consistency is considered one of the most valuable engineering skills developed by CyberLab.

## Closing Statement

CyberLab 2030 is not organized as a traditional course.

# Chapter 09 — Git & Version Control Standards

## Purpose

Version control is a fundamental engineering practice adopted throughout CyberLab 2030.

Every document, script, infrastructure definition, configuration, and source code should be maintained under version control to ensure traceability, reproducibility, collaboration, and continuous improvement.

Git is not treated merely as a backup mechanism, but as the official history of the project's evolution.

---

# Version Control Philosophy

Every meaningful change should be recorded.

Version control provides visibility into the project's evolution, allowing every implementation, correction, improvement, and architectural decision to be tracked over time.

The Git history represents the engineering journey of CyberLab.

---

# Repository Organization

Each repository within the CyberLab ecosystem has a clearly defined responsibility.

Repositories should remain modular, focused, and independent whenever possible.

Every repository should include appropriate documentation, licensing information, and a descriptive README.

---

# Commit Standards

Commits should represent logical units of work.

Each commit should describe a single meaningful change and avoid combining unrelated modifications.

Commit messages should be concise, descriptive, and consistent throughout the project.

CyberLab adopts the Conventional Commits specification whenever applicable.

Examples include:

* feat
* fix
* docs
* refactor
* test
* chore

Clear commit history improves maintainability and project understanding.

---

# Branch Strategy

The default branch represents the stable version of the repository.

Additional branches may be created for new features, experiments, documentation improvements, or infrastructure changes.

Branch names should clearly describe their purpose.

Examples include:

* feature/
* documentation/
* infrastructure/
* automation/
* bugfix/

Branch management should prioritize simplicity and clarity.

---

# Pull Requests

Whenever multiple branches are used, changes should be reviewed before being merged into the main branch.

Pull Requests should clearly describe:

* Objective
* Scope
* Major Changes
* Validation Performed
* Related Documentation

Even in personal projects, structured reviews encourage better engineering discipline.

---

# Versioning

Repositories should evolve through incremental versions.

Major milestones may be identified using semantic versioning principles.

Version numbers should reflect meaningful project evolution rather than the number of commits.

---

# Documentation Versioning

Documentation evolves together with the project.

Significant documentation updates should be committed with the same discipline applied to source code.

Documentation is considered a first-class engineering artifact.

---

# Traceability

Every major implementation should be traceable through:

* Git History
* Commit Messages
* Documentation
* Jira Issues
* Architecture Records

Traceability improves maintainability and facilitates future learning.

---

# Engineering Discipline

Git should become part of the daily engineering workflow.

Frequent commits, descriptive history, organized repositories, and disciplined version control are considered essential professional practices.

The objective is to develop habits consistent with modern software engineering and infrastructure management.

---

## Closing Statement

CyberLab 2030 treats version control as one of its foundational engineering practices.

Every commit represents a documented step in the continuous evolution of the project, transforming Git into both a technical tool and a permanent record of professional growth.


# Chapter 10 — Jira Workflow

## Purpose

CyberLab 2030 adopts an Agile project management approach to organize its learning journey, technical implementations, documentation, and continuous improvement.

Jira serves as the central project management platform, providing visibility, traceability, and structured planning throughout the project's lifecycle.

The objective is not only to manage tasks, but also to develop professional project management habits commonly found in engineering teams.

---

# Project Organization

CyberLab is organized into Epics, Stories, Tasks, and Subtasks.

Each level represents a different level of project planning.

* **Epic** — Represents a major learning module or engineering domain.
* **Story** — Defines a specific capability or learning objective.
* **Task** — Represents an actionable implementation activity.
* **Subtask** — Breaks complex tasks into smaller execution steps when necessary.

This hierarchy ensures clear organization and measurable progress.

---

# Learning Modules

Each Epic represents one complete learning module within CyberLab.

A module is considered complete only after meeting both technical and documentation requirements.

Every module should include:

* Study
* Planning
* Practical Implementation
* Validation
* Documentation
* Lessons Learned
* Retrospective

Learning is measured by completed deliverables rather than completed courses.

---

# Sprint Methodology

CyberLab follows an iterative sprint-based workflow.

Each sprint should have:

* Clearly defined objectives
* Deliverable outcomes
* Technical validation
* Documentation updates
* Sprint retrospective

The sprint duration may vary according to project complexity and personal availability.

Consistency is prioritized over speed.

---

# Definition of Done

A task is considered complete only when all applicable requirements have been satisfied.

These typically include:

* Implementation completed
* Validation performed
* Documentation updated
* Git repository committed
* Jira issue closed
* Lessons learned recorded

Completion represents a finished engineering deliverable rather than a partially completed activity.

Priority should always support the overall CyberLab roadmap.

---

# Traceability

Every Jira issue should be traceable through:

* Git commits
* Documentation
* Architecture diagrams
* Related repositories
* Learning notes

Project management and technical implementation should remain closely integrated.

---

# Continuous Improvement

Sprint retrospectives should identify:

* What worked well
* What can be improved
* Technical challenges encountered
* Documentation improvements
* Process optimizations

Every completed sprint contributes to improving the CyberLab methodology.

---

# Professional Mindset

Jira is not used simply to organize personal tasks.

It is used to simulate the project management practices adopted by modern engineering organizations.

Developing structured planning, prioritization, and execution habits is considered part of the CyberLab learning experience.

---

## Closing Statement

CyberLab 2030 treats project management as an engineering discipline.

Planning, execution, documentation, validation, and continuous improvement are integrated into a single workflow, ensuring that technical growth is accompanied by organizational maturity throughout the entire learning journey.
